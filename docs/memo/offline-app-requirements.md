# PWAオフライン対応化 要件定義書

> **注意**: 本ドキュメントは検討メモであり、実装の一次情報ではありません。実装時は必ずソースコードを正とし、本ドキュメントとの乖離がある場合はソースコードを優先してください。

## 1. 概要

本ドキュメントは、ReactおよびuseSWRを用いたSPA（Single Page Application）におけるPWAオフライン対応の要件を定義する。

## 2. 目的

- ネットワーク切断時でも、IndexedDBに保存されたキャッシュデータを用いて閲覧を継続可能にする。
- オフライン状態であることをユーザーに明示し、誤操作を未然に防ぐことでUX（ユーザーエクスペリエンス）を維持する。

## 3. 対象範囲

- Service Workerによる静的リソースおよび主要画面のプリキャッシュ
- IndexedDBを用いたSWRキャッシュプロバイダの永続化（**GETリクエストのキャッシュのみ対象**）
- オフライン検知（動的しきい値判定）およびUI制御
- オフライン専用ヘルプページとエラーハンドリング

### 対象外

- **オフライン時の書き込み操作（mutate）**: オフライン時の書き込み系操作（POST/PUT/DELETE）は一切サポートしない。下書き保存・未送信キューイング等の仕組みは設けず、オフライン時は書き込み導線を閉じてエラーとする。

## 4. 機能要件

### 4.1 静的リソースと主要画面のプリキャッシュ

- **App Shellのキャッシュ**: アプリの実行に必要なコア資産（HTML, CSS, JS）をキャッシュする。
- **主要画面のプリキャッシュ**: vite-plugin-pwa（Workbox）を用い、インストール時に主要な画面リソースを一括してプリキャッシュすること。これにより、未訪問の画面でもオフライン環境下でUIの表示を可能にする。
  - **補足（技術仕様）**: vite-plugin-pwa の globPatterns 設定を活用し、ビルド時に生成される全ての静的資産をキャッシュ対象に含める。
- **オフライン専用ヘルプページ**: 接続エラー時に遷移・表示するための「ヘルプページ」を常にプリキャッシュ対象に含め、オフライン時でも確実にアクセス可能にすること。

### 4.2 IndexedDBによるSWRキャッシュプロバイダの構築

- **プロバイダの統合**: useSWRのSWRConfigにて、IndexedDBをバックエンドとしたカスタムキャッシュプロバイダを実装する。
- **通信状態による制御**:
  - アプリ側で保持する「オフライン判定フラグ」に連動し、SWRの挙動を制御する。
  - オフライン時は revalidateOnMount や revalidateIfStale 等を false に動的に切り替え、サーバーへの再検証リクエストを抑制し、キャッシュのみを返却すること。
- **有効期限管理**: 保存から6ヶ月を経過したデータは、アプリ起動時に自動削除する。
- **データ型**: キャッシュ対象はテキストベースのAPIレスポンス（JSON等）のみとする。

### 4.3 オフライン時の操作制限とエラーハンドリング

- **GETのみ対応**: オフライン時はキャッシュデータの閲覧（GET）のみをサポートする。書き込み系操作（POST/PUT/DELETE）はオフライン時に一切実行しない。
- **UIによる事前制限**: オフライン状態検知時、書き込み系操作（編集・追加・削除）をグレーアウト（非活性化）し、操作導線を閉じる。
- **一律のエラー対応**: 予期せぬ通信エラーが発生した際は、一律で「エラーモーダル」を表示し、オフライン専用ヘルプページへの導線を提供すること。

### 4.4 オフライン検知とオンライン復帰

- **UI表示**: オフライン状態をインジケーターで明示する。
- **オンライン復帰UI**: ユーザーが手動で接続を再試行できるUI、または自動復帰を試みる仕組みを提供すること。
- **ヘルスチェック**: バックエンドのヘルスチェック用エンドポイントへの通信で復帰を確認する。

### 4.5 接続状態の判定ロジック

`navigator.onLine` とサーバーへの実際の疎通確認を組み合わせた2段階判定を行う。

#### Step 1: `navigator.onLine` による一次判定

- **`navigator.onLine === false` の場合**: 即座に「オフライン」と判定する。
  - ブラウザがOSレベルでネットワークインターフェース（Wi-Fi/LAN）が無効であると認識しているため、通信を試みる必要はない。
- **`navigator.onLine === true` の場合**: まだ「オンライン」とは確定しない。Step 2 へ進む。
  - 公共Wi-Fiのキャプティブポータルやファイアウォール等により、ネットワークインターフェースは有効でも実際の通信ができないケースがあるため。

#### Step 2: サーバー疎通確認（fetch + RTT計測）

`navigator.onLine === true` の場合、軽量なエンドポイント（ヘルスチェックAPI等）に `fetch` を行い、レスポンスの成否とRTT（Round Trip Time）の両方で最終判定する。

- **失敗 (TypeError: Failed to fetch / Network Error)**: 「オフライン」と判定する。
  - ネットワーク遮断、ファイアウォール、認証が必要な公共Wi-Fi等のケースを含む。
- **成功 (200 OK)**: RTTをしきい値と比較して判定する。
  - **RTT ≤ しきい値**: 「オンライン」と確定する。
  - **RTT > しきい値**: 「オフライン」と判定する（通信が極端に遅い場合、実用上オフラインと同等とみなす）。
- **しきい値設計**:
  - 初期判定のしきい値は十分に大きく設定する。
  - オンライン復帰後は、再度オフラインに陥るのを防ぐため、判定しきい値をさらに動的に大きく（緩和）設定すること。

## 5. 非機能要件

### 5.1 データ整合性

- オンライン復帰後、SWRの再検証ロジックにより、キャッシュデータが最新のサーバーデータで適切に更新されること。

### 5.2 ストレージ管理

- **LRU方式による削除**: ストレージ容量が逼迫した際は、古いアクセスデータから削除し、その旨をユーザーに通知すること。
- **PWAインストールの推奨**: iOS等のブラウザによるストレージ自動削除（7日間制限）を回避するため、PWAインストールを促す導線を設ける。

## 2. データ永続化 (IndexedDB / Dexie.js)

### 2.1 データベース・スキーマ設計

Dexie.js を使用し、役割ごとにテーブル（オブジェクトストア）を完全に分離する。

**Database Name**: `AppOfflineDB`

**Tables**:

| テーブル | 用途 |
|---|---|
| `swrCache` | SWRのキャッシュデータ永続化用（GETレスポンスのみ） |
| `userSettings` | アプリケーション設定等（訪問済みTrip一覧など） |

#### swrCache

| フィールド | 説明 |
|---|---|
| `key` | Primary Key (String / Serialized SWR Key) |
| `data` | 取得データ本体 |
| `timestamp` | 保存日時（6ヶ月期限管理用） |
| `lastAccessed` | 最終アクセス日時（LRU容量管理用） |

#### userSettings

| フィールド | 説明 |
|---|---|
| `key` | 設定項目名 |
| `value` | 設定値 |

### 2.2 SWRキーのシリアライズ・ポリシー

- **設計**: SWRから渡されるキー（文字列、配列、オブジェクト）を IndexedDB の主キーとして扱うため、シリアライズ処理を共通化する。
- **実装**: 基本的に `JSON.stringify` を使用するが、オブジェクトのプロパティ順序による不一致を防ぐため、将来的にシリアライズ層でソートを行う拡張性を残す。

### 2.2.1 Dateオブジェクトの永続化に関する注意

SWRキャッシュにはZodスキーマ（`z.coerce.date()`）でパース済みの `Date` オブジェクトが含まれる（例: `Block.startTime`, `Block.endTime`, `Trip.createdAt` 等）。永続化時にデータの型が変化しないよう、以下の方針を守ること。

- **Dexie.js (IndexedDB) の場合**: IndexedDBの内部シリアライズは [Structured Clone Algorithm](https://developer.mozilla.org/en-US/docs/Web/API/Web_Workers_API/Structured_clone_algorithm) を使用しており、`Date` オブジェクトはそのまま保存・復元される。**追加の変換処理は不要**。
- **`JSON.stringify` を経由してはならない**: `JSON.stringify(new Date())` は ISO 8601 文字列に変換され、`JSON.parse` で復元しても文字列のまま `Date` に戻らない。SWRキャッシュのデータ部分（`data` フィールド）を `JSON.stringify` で保存すると、復元後に `Date` が `string` に変化し、コンポーネント側で型不整合や `date.getTime is not a function` 等のランタイムエラーが発生する。
- **実装上の原則**: キャッシュデータの保存・復元には Dexie.js の `put()` / `get()` をそのまま使用し、中間でJSON文字列化を挟まないこと。シリアライズが必要なのはキー（`key` フィールド）のみであり、データ本体はStructured Cloneに委ねる。

### 2.3 SWRカスタムキャッシュプロバイダ (起動時プリロード方式)

起動時に全レコードを一括読み込みするブロッキングロードはデータ量増加に伴い初期表示を劣化させる。一方、完全なオンデマンド遅延ロード方式は `get()` が同期APIであるSWRの仕様と相性が悪く、初回アクセス時に必ずキャッシュミスとなり `mutate` による再注入が必要になるなど実装が複雑化する。

そのため、**起動時に直近アクセスの上位N件をプリロードし、メモリMapに展開する方式**を採用する。プリロード完了までは子コンポーネントのレンダリングをブロックし、完了後は `get()` が常に同期的にキャッシュを返せるようにする。

#### プリロード（hydration）

1. 起動時に `db.swrCache.orderBy('lastAccessed').reverse().limit(MAX_HYDRATE_ENTRIES).toArray()` で直近アクセスの上位エントリをメモリ上の `Map` に展開する。
2. プリロードにはタイムアウト（`HYDRATE_TIMEOUT_MS`）を設け、IndexedDBが遅延した場合でもアプリ起動をブロックしない（`Promise.race` で制御）。
3. プリロード中のエラーは無視し、空のキャッシュで起動を継続する。
4. `SWRProvider` は `hydratePromise` の完了を待ってから子コンポーネントをレンダリングする。

#### 読み込み（get）

- SWRがキャッシュキーを要求した際、メモリ上の `Map` を参照して同期的に返却する。`Map` にないキーは `undefined`（キャッシュミス）を返す。
- プリロード上限を超える古いエントリはメモリに載らないため、オフライン時にはアクセスできない。

#### 書き込み（set）

- SWRがキャッシュを更新する際、メモリ上の `Map` を即時更新し、同時に `db.swrCache.put()` を非同期で呼び出す。UIスレッドをブロックしないよう `await` は行わない（fire-and-forget）。

#### 削除（delete）

- SWRがキャッシュキーを削除した際、`Map` から削除し、`db.swrCache.delete(key)` を非同期で実行する。

### 2.4 クリーンアップ・ロジック

- **時間ベース**: 起動時、`db.swrCache.where('lastAccessed').below(sixMonthsAgo).delete()` を実行。`lastAccessed`（最終アクセス日時）を基準とすることで、頻繁にアクセスされるエントリは保持し、長期間参照されていないエントリのみを削除する。
- **容量ベース (LRU)**: 容量逼迫時、`db.swrCache.orderBy('lastAccessed').limit(N).delete()` により古い順に一定件数を削除。
- **通知**: 削除レコードが存在した場合、React側でフラグを立て、マウント後にトースト等で通知。

## 実装フェーズ

実装は以下の5フェーズに分割し、段階的に進める。各フェーズは前段の成果物に依存するため、原則として順序通りに実施する。ただし、Phase 1 と Phase 2 は独立しているため並行着手が可能。

### Phase 1: データ永続化基盤（IndexedDB + Dexie.js）

**目的**: オフライン時にデータを表示できる土台を作る。

**対応する機能要件**: 4.2 IndexedDBによるSWRキャッシュプロバイダの構築

| # | 作業内容 | 詳細 |
|---|---------|------|
| A-1 | Dexie.js導入 + スキーマ定義 | `AppOfflineDB`（`swrCache`, `userSettings`テーブル）を定義 |
| A-2 | SWRカスタムキャッシュプロバイダ実装 | 遅延ロード方式（メモリMap + IndexedDB非同期読み出し） |
| A-3 | SWRConfig統合 | アプリ全体を`SWRConfig`でラップし、カスタムキャッシュプロバイダを接続 |
| A-4 | useVisitedTripsのIndexedDB移行 | localStorage → `userSettings`テーブルに統一。既存API（`addVisitedTrip`/`removeVisitedTrip`）は維持し、内部ストレージのみ差し替え |
| A-5 | クリーンアップロジック | 6ヶ月期限の自動削除（起動時実行） |

**成果物**: オンライン状態でもAPIレスポンスがIndexedDBに自動保存され、次回起動時にキャッシュから即座に表示される。

**前提**: なし（独立して着手可能）

## 設計決定事項

### D1. SWRオフライン制御方針

- **方針**: `SWRConfig` のグローバル設定で、オフライン時の `revalidateOnMount`, `revalidateIfStale` 等を一括制御する。個別フック（`useTrips`, `usePages`, `useBlocks` 等）の変更は不要。
- **補足: `refreshInterval` の制御**: グローバル `SWRConfig` ではローカル設定の `refreshInterval` を上書きできないため、編集モードのポーリング（5秒間隔）はオフライン atom を参照し、各フック側で `isOffline ? 0 : 5000` のように動的に切り替える。
